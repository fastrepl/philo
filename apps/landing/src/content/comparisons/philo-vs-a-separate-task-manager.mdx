Philo stores daily notes as markdown files on disk, but edits them in memory as TipTap JSON.

## Quick take

Choose Philo when the center of gravity is the note itself. Choose a separate task manager when the center of gravity is somewhere else.

## Comparison table

| Decide this way if... | Philo | A separate task manager |
| --- | --- | --- |
| You want task context to stay attached to the note | Strong fit because planning happens in the note itself. | Often requires jumping between capture and execution surfaces. |
| You want recurring work to rebuild your note automatically | Built around bringing recurring tasks back on schedule inside daily notes. | Possible, but usually outside the note and detached from writing context. |
| You want a dedicated shared task system for a team | Less ideal. Philo is centered on a personal daily planning loop. | Often the better fit. |

## What Philo is actually optimizing for

Daily notes do not inline the full widget payload.

The scope refresh covers the current journal root, vault root, assets directory, Excalidraw directory, and widgets directory.
